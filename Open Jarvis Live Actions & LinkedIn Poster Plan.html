2 · Architecture

The LinkedIn Poster reuses the already-attached Edge session (the same CDP connection Jarvis uses for Gemini) so you are already logged into LinkedIn — no passwords, no second browser. As it acts, it captures a JPEG of the page after each step and streams it to the Hub over Server-Sent Events (SSE). The browser window is also visible on your desktop; the in-app live view mirrors it.

Why screenshots, not video

A per-step JPEG streamed over SSE is robust, dependency-free, and works through the same CDP attach we already use. It's the difference between "watch it click" and a brittle video codec pipeline. Frames are ~30–60 KB at quality 50; one per action plus key waits.

Same engine, two new consumers

We add one tiny export — getEdgeContext() — to the bridge. The LinkedIn Poster and an upgraded Browser Agent both use it + the same live-view component, so "see what it's doing" is consistent everywhere.

4b · Self-correction (sees the screen, fixes itself)

LinkedIn renames classes and reshuffles its DOM all the time, so a fixed selector eventually misses. Instead of failing, the engine captures the DOM and a screenshot and corrects itself through three tiers — each successful recovery shows up live as "🔧 Self-corrected …".

tier 1

Static selectors

Try the known LinkedIn selectors first — fast, zero cost. Works on a normal day.

tier 2

DOM snapshot

If they miss, enumerate every visible interactive element and pick the best match for the intent by text / aria / placeholder. No LLM, instant.

tier 3

Gemini vision

Still stuck? Send Gemini the screenshot + the element list and ask it to point at the right one. The engine literally sees the screen and decides.

need "Start a post"  ─▶  tier1 static  ✗ miss
                          ─▶  tier2 DOM    ▸ snapshot elements ▸ match text="Start a post"  ✓  🔧 self-corrected (DOM)
                          ─▶  tier3 vision ▸ screenshot + list ▸ Gemini: "#3"               ✓  🔧 self-corrected (vision)

7 · Safety & control

You press Post — always

The composer code has no path that clicks Post. It finds the button, screenshots the ready state, and stops. The tab stays open in your Edge for you to review and submit.

Your session, your machine

Runs in your existing signed-in Edge profile over the local CDP port. No LinkedIn password is entered or stored; nothing leaves your machine except the LinkedIn page you'd visit anyway.

Images stay in-project

Only files resolved inside the selected project folder can be attached (path-escape guarded), so the picker can't be tricked into uploading arbitrary files.

Sandboxed canvas

The HTML canvas is a sandboxed iframe served from the same guarded /files/ route used today — no new file exposure.

8 · How it gets e2e-tested

We won't gamble on LinkedIn's live DOM (or risk your real account) in tests. The suite spins up a local mock LinkedIn composer (same selectors: "Start a post", .ql-editor, file input, .share-actions__primary-action) and drives the real composer code against it via the live Edge attach.

Asserted automatically

  • Composer opens; the plan text lands in the editor (typedText matches)
  • Selected image is attached (imageCount ≥ 1)
  • Post button is found but posted = false (never clicked)
  • ≥ 1 live screenshot frame streamed during the run
  • Project image listing returns the reference photo
  • HTML canvas iframe loads the generated page in-app
  • Agent stream emits per-step frames; Hub stays healthy

Real LinkedIn readiness

The same code carries multiple resilient selector fallbacks for LinkedIn's actual composer (Start-a-post trigger, contenteditable editor, hidden file input, Photo button, Post button). Because it stops before Post, you can safely do a real dry run against linkedin.com yourself — it fills, you decide.
