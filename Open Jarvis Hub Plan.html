Open Jarvis Hub — one app, many capabilities

Today's Marketing Studio is one module. This plan turns the local web panel into a capability hub that also surfaces the existing OpenJarvis model — chat, deep research, code assistant, scheduled monitors, morning digest, channels, memory & skills — plus the browser/Webapp-LLM modules we built. Below are the architecture, the module catalog, and rendered mockups of every screen so you can see it before we build it.

Nothing here is built yet — this is the proposal. The two quick wins you asked for (a double-click launcher and an in-UI Approve button) are folded into this hub design (see §Launcher and §Approve).

1 · Vision

One window

A single local app (no terminal) with a left sidebar of capabilities. Each is a "module" you click into.

Two engines underneath

The existing OpenJarvis core (Python, local-first) for chat/research/agents, and the Webapp-LLM browser bridge for tasks that ride your Gemini license + MDM Edge.

Extensible by design

A module registry: adding a capability = one entry + a panel. Marketing, Research, Browser Agent and the core agents all plug in the same way.

The Hub itself stays thin: it renders modules and forwards actions. Local-first work (chat, research, code) goes to the OpenJarvis jarvis CLI; anything that needs your Gemini Webapp license or a real browser goes to the bridge. The remembered jarvis-mdm.config.json (Edge profile + webapp) drives the browser side.

5 · Zero-terminal launcher

A Start Jarvis.bat on your Desktop. Double-click → it starts the hub server and opens your browser at the dashboard. No commands.

@echo off
cd /d "%~dp0tools\playwright-mdm"
start "" http://localhost:4100
node jarvis-ui.js

Optional: a tiny tray/Tauri wrapper later for a true desktop app icon.

What you do

  1. Double-click Start Jarvis.
  2. Browser opens to the dashboard.
  3. Pick a module, click around. Done.

Reminder kept in-app: close Edge before browser-powered runs so Jarvis owns a clean session.

6 · In-UI approval workflow

Every generated item lands in approvals/pending/. The Marketing module shows them with preview + two buttons:

✓ Approve all → moves to approvals/approved/ ✕ Reject → moves to approvals/rejected/ 🔎 Preview each as real HTML

New endpoints: POST /api/approve and POST /api/reject (per-file or all). Nothing publishes without a click — the gate stays human.

7 · Build rollout (phased)

Phase 1 — Hub shell + quick wins

  • Refactor UI into sidebar + module registry
  • Start Jarvis.bat launcher
  • Approve/Reject buttons + endpoints
  • Marketing & Settings modules (already built) slot in

Phase 2 — Bridge modules

  • Ask Jarvis (chat via askGemini / local)
  • Deep Research (research())
  • Browser Agent (act())

Phase 3 — OpenJarvis core

  • Shell out to the jarvis CLI for chat / code / research
  • Monitors & Morning Digest from existing configs
  • Surface local model picker (Ollama)

Phase 4 — Reach

  • Channels (Telegram / Discord / Slack)
  • Memory & Skills browser
  • Optional Tauri desktop wrapper

8 · Decisions (locked)

First modules

Ask Jarvis · Deep Research · Browser Agent · Scheduled Monitors — all four.

Core engine

Both, switchable per module — local jarvis CLI or the Gemini Webapp-LLM bridge.

App form

Browser app now (:4100) → Tauri desktop app as the eventual target.

Approval

Approve-all per run (one button per pipeline run).

Build order locked in: Phase 1 (Hub shell + launcher + Approve-all + bridge modules Ask/Research/Browser Agent) → Phase 2 (per-module engine toggle, wire local jarvis CLI) → Phase 3 (Monitors scheduler + Morning Digest) → Phase 4 (Channels + Tauri desktop wrapper).
